# 程序开发任务清单

本文档用于规划 HijackSoul 的 Godot 框架验证阶段。当前目标不是制作正式内容，而是先完成一套适合“锈湖式 2D 点触解谜体验”的可堆量开发框架。

## 一、框架目标

- 支持 2D 场景/视角切换，例如房间正面、侧面、近景、柜子内部等。
- 支持在 Godot 场景中直接拖拽摆放交互物，位置和层级由场景文件保存。
- 支持交互物 Inspector 配置，尽量接近 UE 蓝图基类的使用体验。
- 支持物品点击、查看、拾取、使用、触发对白、触发事件、打开谜题。
- 支持背包物品选择，并根据当前选中物品触发不同交互结果。
- 支持 NarrRail 负责对白、故事分支和故事事件。
- 支持 NarrRail 事件操控游戏流程，例如解锁视角、给予物品、设置 flag、切换场景。
- 支持自动存档：一次交互动作链执行完成后统一保存。
- 支持后续批量制作房间、物件、谜题和故事流程。

## 二、核心运行循环

```text
玩家点击交互物
-> InteractiveObject 上报点击请求
-> InteractionManager 判断条件和当前背包物品
-> ActionRunner 执行动作链
-> 更新 GameState / Inventory / StoryBridge / SceneFlow
-> UI 播放反馈和提示
-> SaveManager 自动保存
```

![HijackSoul 核心运行循环](./images/核心运行循环.png)

需要避免交互物直接操作其他系统。交互物只描述“自己是什么”和“被点击后请求什么”，实际逻辑交给全局或模块服务处理。

## 三、基础系统任务

### 1. EventBus

- 建立全局事件总线，用于模块间低耦合通信。
- 定义基础事件：
  - 交互物被点击
  - 物品被拾取
  - 背包物品被选中
  - 视角切换请求
  - 故事事件触发
  - 自动存档请求
- 后续原则：模块之间优先通过 EventBus、公开 API 或 Godot signal 通信。

### 2. GameState

- 保存运行时全局状态。
- 第一阶段状态范围：
  - 当前 `view_id`
  - 游戏 flags
  - 已改变状态的交互物
  - 已完成谜题
  - 当前章节/故事进度引用
- 提供统一读写接口：
  - `set_flag(flag_id, value)`
  - `get_flag(flag_id)`
  - `set_object_state(object_id, state)`
  - `get_object_state(object_id)`

### 3. SaveManager

- 存档路径使用 Godot `user://`。
- 不写入 `hijacksoul/data/` 或 `hijacksoul/assets/`。
- 存档内容包含：
  - 当前视角
  - 背包物品
  - 当前选中物品
  - GameState flags
  - 交互物状态
  - 谜题状态
  - NarrRail save snapshot
- 交互动作链完成后再统一自动保存，避免保存中间状态。
- 提供手动保存、自动保存、读档、清档接口。

### 4. SceneFlowManager

- 负责视角切换和场景加载。
- 每个视角建议对应一个 Godot scene：

```text
levels/chapter_01/room_01_front.tscn
levels/chapter_01/room_01_left.tscn
levels/chapter_01/room_01_drawer_closeup.tscn
```

- 维护 `view_id -> scene_path` 映射。
- 支持黑场遮罩、异步加载 Loading、切换完成事件。
- 切换完成后通知交互物从 GameState 恢复状态。

### 5. InteractionManager

- 接收 `InteractiveObject` 的交互请求。
- 判断：
  - 交互物是否启用
  - 交互物是否可见
  - 当前是否选中了背包物品
  - 是否满足 flag 条件
  - 是否存在该物品对应的交互分支
- 选择合适动作链交给 ActionRunner。
- 如果没有有效交互，触发默认反馈，例如普通查看对白或无效使用提示。

### 6. ActionRunner

- 顺序执行交互动作链。
- 第一阶段支持动作类型：
  - `show_dialogue`：显示普通文本或请求 NarrRail 对白
  - `emit_story_event`：向 StoryBridge 发送故事事件
  - `set_flag`：修改 GameState flag
  - `pickup_item`：加入背包并隐藏场景物件
  - `remove_item`：从背包移除物品
  - `change_view`：切换到指定视角
  - `set_object_state`：改变交互物状态
  - `play_animation`：播放交互物或当前视角动画
  - `open_puzzle`：打开谜题界面
  - `show_toast`：显示获得物品或提示
- 动作链完成后触发自动存档。

### 7. InventoryManager

- 管理背包物品列表。
- 管理当前选中物品。
- 支持拾取、移除、选中、取消选中。
- 向 UI 发送背包变化事件。
- 与 InteractionManager 对接，用当前选中物品判断交互分支。

### 8. StoryBridge

- 负责游戏框架和 NarrRail 的双向桥接。
- 游戏到 NarrRail：
  - 交互物点击触发 `emit_story_event`
  - 谜题完成触发故事事件
  - 关键物品获得触发故事事件
- NarrRail 到游戏：
  - 收到 NarrRail event 后转换为 ActionRunner 动作
  - 支持设置 flag、给予物品、移除物品、切换视角、解锁交互物、打开 UI
- StoryBridge 保存和恢复 NarrRail snapshot。
- 场景物件不直接调用 NarrRail。

## 四、场景与视角框架

### 1. RoomView 基类

- 每个视角 scene 的根节点使用统一脚本或统一结构。
- 建议节点结构：

```text
RoomView
  Background
  ObjectLayer
    InteractiveObject_Clock
    InteractiveObject_Drawer
    InteractiveObject_Photo
  ForegroundLayer
  ExitLayer
    Exit_Left
    Exit_Right
  Camera2D
```

- RoomView 负责：
  - 注册当前视角 ID
  - 收集子节点交互物
  - 通知交互物恢复状态
  - 处理进入视角时的默认事件

### 2. 视角切换点

- 方向按钮、门、近景入口都可以复用 InteractiveObject。
- 配置动作为 `change_view`。
- 支持 hover 提示，例如“看向左侧”“查看抽屉”。

### 3. 场景生产规则

- 位置和层级在 Godot scene 内拖拽调整。
- 每个可存档物件必须有稳定唯一 `object_id`。
- 建议 ID 格式：

```text
chapter_01.room_01.drawer
chapter_01.room_01.clock
chapter_01.room_01.photo
```

## 五、交互物框架

### 1. InteractiveObject 基类

- 建立可复用基础场景：

```text
hijacksoul/modules/interaction/interactive_object.tscn
hijacksoul/modules/interaction/interactive_object.gd
```

- 基础节点建议：

```text
InteractiveObject
  Sprite2D
  Area2D
    CollisionShape2D
  Highlight
  AnimationPlayer
```

- 第一阶段先做一个通用交互物基类，后续再按需要派生：
  - PickupObject
  - ViewExitObject
  - PuzzleEntryObject
  - StoryTriggerObject

### 2. Inspector 配置字段

- `object_id`
- `display_name`
- `hover_text`
- `default_state`
- `save_enabled`
- `visible_condition`
- `enabled_condition`
- `default_interactions`
- `item_interactions`
- `story_event_on_click`
- `pickup_item_id`
- `target_view_id`
- `puzzle_id`

字段可以先用 Godot exported variables 实现。复杂动作链后续可升级为 Resource 配置。

### 3. 交互分支

需要支持同一个物件根据当前背包物品触发不同结果：

```text
空手点击抽屉 -> 显示“锁住了”
选中钥匙点击抽屉 -> 消耗钥匙，设置 drawer_unlocked，切换抽屉打开状态
抽屉打开后点击 -> 进入抽屉近景
```

### 4. 状态恢复

- 场景加载后，交互物根据 `object_id` 从 GameState 查询状态。
- 可恢复内容：
  - visible
  - enabled
  - state
  - picked
  - animation_state
- 已拾取物品重新进入场景时应保持隐藏。

## 六、NarrRail 对接任务

### 1. 对白播放

- 接入 NarrRailSession 或 NarrRailOutlineRunner。
- StoryBridge 接收 `line_changed`、`choices_changed`、`ended`、`event_emitted`。
- UI 显示对白文本、说话人、选项。

### 2. 故事事件驱动游戏

- 定义 NarrRail event 到游戏 action 的映射。
- 第一阶段支持：
  - `set_flag`
  - `give_item`
  - `remove_item`
  - `change_view`
  - `unlock_object`
  - `lock_object`
  - `open_puzzle`
  - `show_toast`

### 3. 游戏事件推进故事

- 交互物可配置触发故事事件。
- 谜题完成可触发故事事件。
- 背包获得关键物品可触发故事事件。

### 4. 存档整合

- SaveManager 保存 NarrRail snapshot。
- 读档时先恢复故事 session，再恢复游戏状态和当前视角。

## 七、背包与拾取系统

- 建立背包 UI 面板。
- 支持物品图标、名称、选中态。
- 支持拾取提示条：

```text
右下角显示“获得【物品名】”，1.5 秒后自动消失
```

- 选中背包物品后点击场景物件，InteractionManager 根据物品 ID 选择交互分支。
- 无效使用时给出统一反馈。

## 八、谜题/小游戏框架

- 第一阶段只做入口和完成回调，不做复杂谜题系统。
- 谜题以独立 scene 实现。
- 统一通过 `open_puzzle(puzzle_id)` 打开。
- 谜题完成后触发：
  - `puzzle_completed(puzzle_id)`
  - 设置 GameState flag
  - 可选触发 NarrRail event
  - 自动保存
- 谜题状态进入存档。

## 九、演出与 UI

### 1. 场景切换

- 黑场遮罩。
- Loading 图标。
- 切换目标提示，例如“走廊”“抽屉内部”。

### 2. 交互反馈

- 鼠标悬停高亮。
- 名称 Tooltip。
- 点击无效反馈。
- 拾取提示条。

### 3. 对白 UI

- 显示 NarrRail 对白。
- 支持点击继续。
- 支持选项列表。
- 支持对白中触发事件。

### 4. 调试 UI

- 显示当前 view_id。
- 显示当前 flags。
- 显示背包物品。
- 显示最近触发事件。
- 支持一键保存、读档、清档。

## 十、2D 动画技术方案

- 普通物件状态变化优先使用 AnimationPlayer。
- 简单 hover 高亮使用材质、调色或额外 Highlight 节点。
- 场景内一次性演出可放在 RoomView 的 AnimationPlayer。
- 交互物长期状态必须能从 GameState 恢复，不能只依赖播放过的动画。
- 动画结束后如需改变逻辑状态，由 ActionRunner 或交互物回调写入 GameState。

## 十一、最小验证 Demo

第一阶段验收目标：做一个可完整跑通、可存档恢复的小样板。

### Demo 内容

- 一个房间，三个视角：
  - 正面视角
  - 左侧视角
  - 抽屉近景
- 一个可查看物品：
  - 点击触发 NarrRail 对白
- 一个可拾取物品：
  - 点击进入背包
  - 场景中隐藏
  - 自动存档
- 一个需要背包物品打开的交互物：
  - 空手点击提示锁住
  - 选中钥匙点击后打开
  - 状态可存档恢复
- 一个谜题入口：
  - 打开独立谜题 scene
  - 完成后设置 flag
  - 触发 NarrRail 事件
- 一个故事分支：
  - NarrRail 根据事件或变量显示不同对白

### Demo 验收标准

- 退出游戏后读档能回到正确视角。
- 已拾取物品不会重新出现。
- 背包物品正确恢复。
- 抽屉打开状态正确恢复。
- NarrRail 对白进度和变量正确恢复。
- 谜题完成状态正确恢复。
- 所有核心交互都通过通用框架完成，不为单个物件写特殊逻辑。

## 十二、阶段拆分

### 阶段 1：基础闭环

- 建 EventBus、GameState、SaveManager 骨架。
- 建 SceneFlowManager 和 RoomView 基类。
- 建 InteractiveObject 基类。
- 实现点击、hover、tooltip、基础 action。
- 实现一个视角内的查看和拾取。

### 阶段 2：背包与物品使用

- 建 InventoryManager。
- 建背包 UI。
- 支持选中物品点击交互物。
- 支持物品消耗和无效使用反馈。

### 阶段 3：视角切换与状态恢复

- 实现多个 RoomView 切换。
- 交互物按 object_id 恢复状态。
- 实现场景切换黑场和 loading。

### 阶段 4：NarrRail 桥接

- 接入 NarrRail 对白 UI。
- 游戏交互触发 NarrRail event。
- NarrRail event 转成游戏 action。
- 存档整合 NarrRail snapshot。

### 阶段 5：谜题框架

- 实现 open_puzzle 统一入口。
- 实现谜题完成回调。
- 谜题状态进入 GameState 和存档。

### 阶段 6：生产工具与规范

- 完善交互物 Inspector 配置体验。
- 增加重复 object_id 检查。
- 增加缺失资源、缺失 action、无效 view_id 检查。
- 增加调试 UI。
- 编写场景制作规范。

## 十三、暂缓事项

- 不制作正式剧情内容。
- 不制作正式美术资产。
- 不实现复杂 AI、战斗、伤害公式。
- 不过早设计大量谜题类型。
- 不强制 Excel 驱动场景摆放；场景位置和层级优先由 Godot scene 保存。
